# oxide-matmul — Deep Analysis

A head-to-head benchmark of two cuda-oxide kernels against an nvcc CUDA C++ baseline, isolating the cost of Rust's slice bounds checks on a naive 4096×4096 f32 matrix multiply.

## 1. Test Description

The benchmark runs the textbook, unblocked, non-tiled matrix multiply

```
C[r, c] = Σ_k A[r, k] * B[k, c]
```

over square 4096×4096 f32 matrices in global memory, with a `(16, 16, 1)` thread block and a `(256, 256, 1)` grid. Each thread computes exactly one output element by walking `k = 0..4096` and accumulating into a register. There is no shared-memory tiling, no vectorized loads, no register blocking, and no unrolling hints — this is intentional.

Why **naive** matmul: we are not benchmarking cuBLAS-class kernels, we are benchmarking the code-generation quality of three compilers (cuda-oxide via LLVM NVPTX, nvcc, and WGSL/naga) on a common reference algorithm. Tiled kernels would hide compiler differences behind hand-written shared-memory pipelines.

Why **4096×4096**: it pushes past L2 (≈64 MB of working set for the three matrices), produces 137.44 GFLOP per iteration — large enough that per-launch overhead is <1% of wall time — and is a multiple of the 16×16 block, so no tail handling pollutes the inner loop.

Why **two Rust variants**: the kernel is compiled twice from almost-identical sources. `matmul` uses safe slice indexing (`a[r*dim_us + k]`), which lowers to panic-on-out-of-bounds. `matmul_unchecked` uses raw-pointer arithmetic (`*a_base.add(r*dim_us + k)`). Everything else — block shape, accumulator type, store, early-out on `row/col >= dim` — is identical. The delta between the two isolates exactly one variable: the per-iteration bounds-check predicate generated by rustc's safe-indexing lowering.

## 2. Methodology

Each kernel (safe and unchecked) runs **1 warmup launch + 10 timed iterations**. The warmup launch occurs before the timed loop and is `stream.synchronize()`d to pay JIT compile, module load, and cache-warm costs up front. Timing uses `std::time::Instant::now()` bracketing a `cuda_launch!` + `stream.synchronize()`, i.e. **CPU wall clock**, not `cudaEventRecord`.

Wall-clock was chosen deliberately: we want the cost a caller actually observes (launch latency + kernel body + sync), not the pure device-side body. For a ~23 ms unchecked kernel and ~57 ms safe kernel, launch overhead (~5–50 µs) is well below 1% of the measurement and does not influence ranking. `cudaEvent`s would shave that sub-1% off both numbers equally and change no conclusion.

We report both **best** and **median** of the 10 samples. The median is the headline figure — it is robust to a single cold-cache or scheduler-blip outlier (iter 4 and iter 8 on the safe kernel drift to ~60–62 ms from a ~55 ms base, see `run.log`). The best illustrates steady-state floor.

Inputs are **deterministic and fixed**: `a[i] = (i % 7) * 0.01_f32` and `b[i] = (i % 11) * 0.01_f32`. Coprime moduli (7 and 11) ensure non-trivial products across the reduction. All values are small (<0.07) so the 4096-wide accumulation stays within f32 precision and the spot-check tolerance of 1e-3 relative error is meaningful.

Correctness is verified after the benchmark by recomputing the dot product on the host at three spot indices: `(0,0)`, `(100,100)`, `(4095,4095)` — hitting the top-left, interior, and bottom-right corner of C. The latest run reports `3/3`.

## 3. Results

4096×4096 f32 matmul, 137.44 GFLOP/iter, 16×16 block, RTX 5090 / sm_120 / WSL2 / driver JIT to native.

| impl                         | best ms | median ms | TFLOPS (med) | vs nvcc |
|------------------------------|--------:|----------:|-------------:|--------:|
| nvcc CUDA C++ `-arch=sm_89`  |   20.64 |     21.53 |         6.39 |   1.00× |
| cuda-oxide **unchecked**     |   22.43 |     23.15 |         5.94 |   0.93× |
| cuda-oxide **safe** (slices) |   51.82 |     56.86 |         2.42 |   0.38× |
| wgpu / WGSL via llvmpipe CPU |  ~25000 |    ~25700 |        0.005 | ~0.001× |

Oxide safe-vs-unchecked median speedup: **2.46×**. All three GPU paths pass the 3/3 spot check.

## 4. PTX Deep-Dive

Both kernels live in `oxide_matmul.ptx`. The inner loops are:

**`matmul_unchecked`** (lines 54–67) — one bounds check, one load pair, one mul+add per iteration:

```ptx
$L__BB0_3:                              // =>This Inner Loop Header
    setp.ge.u64     %p3, %rd24, %rd1;   // k >= dim ?
    @%p3 bra        $L__BB0_5;          //   yes -> store & exit
    ld.b32          %r13, [%rd22];      // A[r*dim + k]
    ld.b32          %r14, [%rd23];      // B[k*dim + c]
    mul.rn.f32      %r15, %r13, %r14;
    add.rn.f32      %r16, %r16, %r15;   // acc += a*b
    add.s64         %rd24, %rd24, 1;    // k++
    add.s64         %rd23, %rd23, %rd4; // B ptr += dim*4
    add.s64         %rd22, %rd22, 4;    // A ptr += 4
    bra.uni         $L__BB0_3;
```

One `setp` per iteration — that is the loop-exit test `k < dim_us`, nothing more.

**`matmul`** (safe, lines 124–147) — **three** `setp.ge.u64 ... @%p bra` predicates per iteration:

```ptx
$L__BB1_3:                              // =>This Inner Loop Header
    setp.ge.u64     %p3, %rd30, %rd1;   // (1) k >= dim_us (loop exit)
    @%p3 bra        $L__BB1_7;
    add.s64         %rd23, %rd5, %rd30;
    setp.ge.u64     %p4, %rd23, %rd16;  // (2) r*dim+k >= a.len  ->  panic
    @%p4 bra        $L__BB1_9;          //     exits via inline asm `exit;`
    setp.ge.u64     %p5, %rd28, %rd18;  // (3) k*dim+c >= b.len  ->  panic
    @%p5 bra        $L__BB1_10;         //     exits via inline asm `exit;`
    ld.b32          %r4,  [%rd27];
    ld.b32          %r14, [%rd29];
    mul.rn.f32      %r15, %r4, %r14;
    add.rn.f32      %r16, %r16, %r15;
    ...
    bra.uni         $L__BB1_3;
```

`setp.ge.u64 dst, a, b` sets predicate `dst = (a >= b)` on 64-bit operands, and `@%p bra L` is a predicated branch (“branch to L if p is true”). The targets `$L__BB1_9` and `$L__BB1_10` both contain a single inline-asm `exit;` — that is rustc's slice-panic path lowered for PTX, i.e. "we would call `core::panicking::panic_bounds_check` on the host, here we just terminate the thread."

## 5. Why Safe Is 2.46× Slower

The kernel body guards the *output* with `if row >= dim || col >= dim { return; }` at the top, which already proves `r < dim` and `c_idx < dim`. Mathematically this is enough to prove every inner-loop access is in bounds, because `a.len() == b.len() == dim*dim`. **The Rust compiler does not know that.** `a` and `b` arrive as `&[f32]` with opaque runtime lengths; the `dim` parameter is a separate `u32`; there is no dependent-types link between them.

So the lowering of `a[r * dim_us + k]` stays faithful to Rust semantics: compute the index, compare against `a.len()` (materialized in `%rd16`), and branch to a panic handler if out of range. Same for `b[k * dim_us + c_idx]` against `%rd18`. That is predicates `%p4` and `%p5` above. They fire **every iteration** for 4096 iterations per thread × 16.7 M threads = ~68 B extra compare/branch pairs per launch.

The practical cost is not just the two extra instructions in ALU-issue count. They also:
1. Double the register pressure on the loop-carried 64-bit index vs. length values (`%rd16`, `%rd18`, `%rd23`, `%rd28`), reducing occupancy headroom.
2. Push the inner loop past the NVPTX backend's simple-loop heuristics, so it is never unrolled.
3. Serialize with the `ld.b32` loads — the loads are only issued *after* the two predicates prove safe, so the compiler cannot hoist them and cannot overlap them with the previous iteration's `mul/add`.

Removing the bounds checks (unchecked kernel) collapses the per-iteration work from "3 setp + 2 branches + 2 loads + mul + add + 4 pointer bumps" to "1 setp + 1 branch + 2 loads + mul + add + 3 pointer bumps". The 2.46× factor matches the arithmetic density change almost exactly.

## 6. Why Unchecked Is Still ~7% Behind nvcc

From `analysis/ptx-stats.txt`:

```
oxide_matmul.ptx (cuda-oxide):   fma.rn: 0   mul.rn+add.rn: 4   ld.global.nc: 0   setp: 8   lines: 164
nvcc matmul.ptx:                 fma.rn: 5   mul.rn+add.rn: 0   ld.global.nc: 10  setp: 7   lines: 124
```

Two code-generation features nvcc uses that the LLVM NVPTX backend does **not** emit here:

**(a) No `fma.rn.f32` contraction.** The oxide inner loop emits `mul.rn.f32` followed by `add.rn.f32` as two separate instructions. nvcc contracts the same pattern into a single fused multiply-add. On Ada / Blackwell, FMA is one issue slot and one rounding; the two-op sequence is two issues and two roundings. This is worth a few percent directly, plus it halves the FP pipe pressure in the hot loop.

**(b) No `ld.global.nc` (read-only / `__ldg`) cache path.** nvcc emits 10 `ld.global.nc` loads because the C++ kernel declares `A` and `B` as `const float* __restrict__`; the compiler proves they are read-only and non-aliasing and routes their loads through the read-only data cache. Rust's `&[f32]` carries no such annotation in the LLVM IR the NVPTX backend sees, so loads go through the default L1/L2 path. On memory-bound naive matmul this matters.

**(c) Unrolling.** nvcc's 124-line PTX (vs 164 for oxide) despite doing 4× manual unroll on the reduction suggests tighter scheduling inside each block as well.

Together these account for the residual ~0.93× (≈7%) gap between `matmul_unchecked` and the nvcc baseline. They are compiler-infrastructure issues, not Rust language issues — an `fma` pattern-matcher and an `__ldg` hint for read-only slices would close most of the remaining gap.

## 7. Reproducing

Prerequisites:
- **LLVM 21** toolchain on PATH (`llvm-config --version` → 21.x).
- **Rust nightly** (`rustup default nightly`).
- **cargo-oxide** installed from the cuda-oxide repo: `cargo install --path crates/cargo-oxide`.
- `cuda-oxide doctor` passes (CUDA toolkit, driver, and NVPTX target visible).

Run:

```bash
cd /home/codeseys/cuda-oxide-bench/oxide-matmul
cargo oxide run oxide-matmul
```

This emits `oxide_matmul.ptx` (module) and prints the per-iteration log plus the summary table. Output matches `run.log` within per-iteration jitter. The nvcc baseline is reproduced with `cd ../cuda-matmul && make run`; the WGSL/llvmpipe path with `cd ../wgpu-matmul && cargo run --release`.

## 8. Limitations & Threats to Validity

- **Wall-clock timing includes kernel launch overhead** (~5–50 µs of driver/runtime work per launch on Linux/WSL2). At ~23 ms per iteration this is <0.25% — invisible at the reported precision, but it would dominate if the problem shrank below ~64×64.
- **Single-GPU, single-machine run** on one RTX 5090 under WSL2. WSL2 adds a Hyper-V driver hop that is stable but not identical to bare-metal Linux; absolute TFLOPS may shift ±5% on native Linux. Relative ordering is robust.
- **No `nsys` / `ncu` profiling.** We have not measured achieved occupancy, memory-throughput, or warp-stall reasons directly. The PTX-level explanation of the 2.46× gap is consistent with the code, but not ground-truthed against SM counters.
- **No occupancy tuning.** Block shape `(16,16)` was chosen for parity with the nvcc and WGSL reference kernels, not for optimality on sm_120. A different block shape or a `__launch_bounds__` equivalent could shift all three GPU numbers — but would do so roughly equally and should not change the ratios.
- **JIT compile of sm_80 PTX to native sm_120.** The `.target sm_80` directive at the top of `oxide_matmul.ptx` means the driver JIT-recompiles to the actual Blackwell ISA at first launch. The warmup iteration absorbs this; nothing in the timed loop observes it. nvcc's baseline targets `sm_89`, which also JITs forward.
- **f32 numerical drift is not measured.** The 3-point spot check at 1e-3 relative tolerance confirms bit-level-equivalent algorithmic behavior, not per-element accuracy. A full `‖C_oxide − C_nvcc‖_∞` would be stronger but is out of scope.
